# Native Internationalization (i18n)

A world map of history shouldn't be restricted to English readers only. The CodeOrigins ecosystem has a built-in asynchronous dictionary to ensure access to the "Big Data" of software history descriptions in as many languages as possible.

## How `localeLoader.ts` Operates

Unlike systems like `react-i18next` that overload the application with pre-cached dictionaries of thousands of keys, we opted for an extremely polished native fetcher (asynchronous search engine):

Dictionaries are loaded **dynamically** only when the Planet is selected and the screen is about to open, or at the zero boot moment for HUD elements:

1. The `src/lang/` folder has segmented JSON dictionaries (Ex: `en.json`, `pt.json`, `es.json`).
2. Such dictionaries have sub-keys like `hud`, `settings`, but the vast majority of the volume is found in the languages block itself (`languages.python.description`).
3. The base service uses generic TypeScript to hit these JSON matrices at Runtime via Webpack/Vite dynamic imports: `import("./lang/${locale}.json")`.

> [!TIP]
> This keeps the Base Weight of the initial bundle (`bundle.js`) minimal, pulling translations via Network ONLY lazily and allocating them to the Zustand state when requested for the first time.

## Translating New Pieces of History

To help compose language descriptions, a global contributor in CodeOrigins will only need to follow the standardized planet object within their respective language package. The system is passive to new additions.

```json
// en.json
{
  "languages": {
    "my-new-planet": {
      "name": "My New Planet",
      "description": "Language x created for y in the Z years...",
      "paradigm": "Prototype-Oriented"
    }
  }
}
```

If a requested key is not found in a Brazilian voyager's `pt.json`, the system will automatically *fallback* and render the existing block in the underlying English language (`en`) so buttons or details never show brute "String Not Found" bugs.
